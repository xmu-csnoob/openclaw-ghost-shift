# Agent 化名规范 (Agent Pseudonym Convention)

> Ghost Shift 的虚拟员工身份系统——让每个 session 有温度但不泄露信息。

## 设计理念

/live 页面是一个"虚拟办公室"，每个 agent 是里面的"员工"。对外展示时，用化名代替真实技术元数据，营造有人情感的办公氛围，同时避免信息泄露。

## 化名构成

```
{昵称} · {模糊职责}
```

示例：`橘子酱 · 正在理顺一些逻辑`

### 名字生成

**由大模型自由生成**，不使用预定义名字池。唯一约束：

- 暖色调风格（橙、红、琥珀、焦糖等意象）
- 2-3 个字
- 有亲和力、有人味儿
- 同一时刻活跃 session **不允许重名**（取新名时检查现有化名）

### 职责描述

描述自己在干什么，**必须模糊化**，约束：

- ❌ 不含文件名、路径、技术栈、项目名
- ❌ 不含具体 bug/error 信息
- ❌ 不含 API key、token、凭证
- ✅ 用日常语言描述大致活动

### 生成 prompt 模板

```
你是一个虚拟办公室的员工，请给自己取一个化名和职责描述。

规则：
- 名字：2-3字，暖色调风格（橙红琥珀焦糖等意象），可爱/亲切/有趣
- 职责：用日常语言描述你在做什么，不含任何技术细节
- 格式：{名字} · {职责}
- 已有人在用的名字（不能重复）：{existingNames}

你正在做的真实任务（仅供参考取职责风格，禁止泄露）：
{taskDescription}
```

## 虚拟部门

真实 zone → 公开部门名称映射：

| 真实 zone | 公开部门 |
|----------|---------|
| code-studio | 产品创新部 |
| chat-lounge | 客户体验部 |
| ops-lab | 基础架构部 |
| unknown/default | 综合管理部 |

## 前端展示规范

- /live 页面：展示化名 + 模糊职责 + 公开部门
- Hover 信息卡：展示化名、当前状态、简短动态
- **禁止**在公开页面展示：session ID、真实任务描述、文件路径、工具名称

## API 安全规则

1. 公开 API 只返回脱敏后的 PublicAgentDisplay
2. 真实元数据 API 需要 auth token
3. 化名不可逆——不能从化名反推真实 session/任务

## Subagent 命名流程

```
Javis 派出 agent
  ↓
生成化名（大模型取名 + 检查重名）
  ↓
sessions_spawn(label: "{化名}·{职责}", ...)
  ↓
/live 页面展示化名
  ↓
agent 完成 → 化名随 session 结束释放
```
